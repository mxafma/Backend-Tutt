Especificación Técnica Backend MVP
Sistema de Compras para Verdulería
Documento preparado para generación de backend con Spring Boot
Versión: 1.0 | Alcance: MVP backend | Stack: Spring Boot + REST API + JWT + base de datos relacional
1. Objetivo del backend
Crear el backend central de un sistema digital de órdenes de compra para una verdulería que actualmente trabaja con una aplicación básica en Python, documentos impresos y planillas Excel.
El backend debe exponer una API REST que permita a una app Android y a una aplicación web gestionar órdenes de compra, productos, proveedores, compras externas, recepción de mercadería, cálculo de precios sugeridos, reportes básicos y sincronización offline.
2. Alcance del MVP backend
2.1 Incluido
Autenticación con JWT y control por roles.
Gestión básica de usuarios.
Gestión de productos, categorías y proveedores.
Creación y edición de órdenes de compra.
Flujo de estados de la orden: borrador, lista para comprar, en compra, comprada, recibida, cerrada y cancelada.
Detalle de orden con productos, formato, cantidades, costos, factura, comentarios y estado del producto.
Registro de productos agregados en el mercado.
Recepción de mercadería y corrección de datos.
Cálculo de precios sugeridos.
Compras externas recibidas en el local.
Pendientes de compra registrados durante el día.
Reportes básicos administrativos.
Exportación de orden a Excel.
Endpoints mínimos para sincronización offline desde Android.
2.2 Fuera del MVP
Integración automática con Eleventa.
Stock real.
Merma exacta.
Gráficos avanzados.
Adjuntar fotos de documentos.
Notificaciones avanzadas.
Comparación entre precio sugerido y precio real vendido.
Auditoría completa de cada campo modificado.

7. Reglas de negocio del backend
7.1 Flujo de orden
BORRADOR -> LISTA_PARA_COMPRAR -> EN_COMPRA -> COMPRADA -> RECIBIDA -> CERRADA
El comprador debe ver solo la orden activa del día.
Al iniciar la compra, la orden pasa a EN_COMPRA.
Al finalizar compra, la orden pasa a COMPRADA.
Recepción puede corregir cantidades, costos, cantidad interna y comentarios.
Al cerrar, la orden queda en historial y no debe ser editada por usuarios operativos.
El admin puede corregir registros si es necesario.
7.2 Cálculo de precios
Si tieneFactura = true:
  costoConsiderado = costoUnitario
Si tieneFactura = false:
  costoConsiderado = costoUnitario * 1.19
precioSugerido = costoConsiderado / (1 - margen)
margenResultante = ((precioVenta - costoConsiderado) / precioVenta) * 100
costoUnitario = costoTotal / cantidadInterna
Redondeo inicial:
  redondear hacia abajo a la decena más cercana.
  Ejemplo: 1875 -> 1870
7.3 Margen
Margen general inicial: 40%.
El margen general debe ser configurable.
Cada producto puede tener margen recomendado propio.
Si el producto tiene margen propio, usar ese margen como primera sugerencia.
El usuario de recepción puede modificar el precio sugerido y el sistema debe mostrar el margen resultante.

9. Sincronización offline para Android
El backend debe soportar un flujo offline básico para que la app Android pueda trabajar sin conexión en el mercado.
9.1 Bootstrap offline
Endpoint sugerido: GET /api/sync/bootstrap
Debe entregar al comprador los datos mínimos para trabajar offline:
usuario actual
orden activa
detalle de la orden
productos básicos
categorías
proveedores/lugares básicos
parámetros de margen y redondeo
9.2 Envío de cambios offline
La app Android guardará cambios localmente y los enviará cuando vuelva internet. El backend debe aceptar cambios por lote.
POST /api/sync/cambios
Payload sugerido:
{
  "deviceId": "android-001",
  "usuarioId": 2,
  "cambios": [
    {
      "tipo": "DETALLE_ORDEN_ACTUALIZADO",
      "entidadId": 15,
      "fechaLocal": "2026-05-18T10:30:00",
      "data": { ... }
    }
  ]
}
9.3 Regla de conflicto MVP
Si dos usuarios modifican el mismo dato, gana el último cambio recibido/procesado. Esta regla es simple para el MVP. Más adelante puede agregarse resolución de conflictos.
10. Reportes básicos del MVP
Gasto diario en compras.
Compras del día.
Compras con factura.
Compras sin factura.

Productos no comprados.
Cantidad comprada por producto.
Los reportes solo estarán disponibles para ADMIN en la primera versión. La visualización inicial será mediante tablas simples.
11. Validaciones mínimas
No permitir crear orden sin fecha planificada.
No permitir pasar a LISTA_PARA_COMPRAR si la orden no tiene productos.
No permitir finalizar compra sin advertir productos pendientes.
Permitir cerrar con productos pendientes, pero registrando la situación.
No permitir costos negativos.
No permitir cantidad interna cero si se debe calcular costo unitario.
Permitir producto libre cuando el comprador agrega algo no registrado.
Solo ADMIN puede eliminar registros.
Comprador solo puede consultar la orden activa.
